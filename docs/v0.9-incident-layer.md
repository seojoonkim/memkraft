# MemKraft v0.9.0 — Incident Memory Layer

> Status: **Release Candidate 1** (`0.9.0-rc1`) · no breaking changes vs v0.8.x
> Track record: v0.8.4 → v0.9.0-rc1 adds 3 mixins, 6 new APIs, 66 new tests.

## TL;DR

MemKraft 0.9 extends the "compound knowledge system" from "people / orgs / projects" memory into **operational sense-making**:

- **Incidents** — first-class memory for "what broke and how we fixed it."
- **Runbooks** — reusable symptom → cause → fix patterns, matched on recurrence.
- **RCA** — heuristic scoring of hypotheses, enriched with related incidents and matching runbooks.

All new APIs are **purely additive**. The v0.8 surface (`track`, `update`, `search`, `tier_set`, `fact_add`, `log_event`, …) is untouched.

## Why an "Incident Memory Layer"?

Three concurrent 2026-04 releases (OpenSRE, Minions, Multica) attacked the same surface from different angles. OpenSRE in particular makes the case that **operational events deserve first-class memory**: evidence-backed, runbook-linked, bitemporal.

MemKraft already had the substrate — bitemporal facts, tier/decay, wiki-link graph — it just didn't expose incident shapes on top. v0.9 closes that gap:

- **bitemporal** → incident `detected_at` / `resolved_at` are `valid_from` / `valid_to`.
- **tier/decay** → open incidents auto-tier `core` (always in working set); resolved → `archival`.
- **link graph** → runbooks reference source incidents via wiki-ish ids.

## New APIs

### `incident_record(title, symptoms, …) → incident_id`

```python
iid = mk.incident_record(
    title="gateway double start",
    symptoms=["Address already in use", "pgrep returns 2 pids"],
    severity="high",           # low | medium | high | critical
    affected=["gateway"],
    hypothesis=["stale PID file"],
    evidence=[{"type": "log", "path": "/var/log/gateway.log"}],
    source="cron-log",
    tags=["prod"],
)
```

Validations:
- `title` non-empty, `symptoms` non-empty list, `severity` in the four-way enum.
- Collisions on auto-generated ids get `-2`, `-3` suffixes.
- If `resolution` is passed at creation time, the incident is born resolved and tiered `archival`.

### `incident_update(incident_id, …) → frontmatter dict`

One function, many shapes:

```python
mk.incident_update(iid, add_hypothesis=["race in init"])
mk.incident_update(iid, reject_hypothesis=["cron dup"])       # marks [rejected @ ts]
mk.incident_update(iid, confirm_hypothesis=["race in init"])  # marks [confirmed @ ts]
mk.incident_update(iid, add_evidence=[{"type":"metric","v":0.99}])
mk.incident_update(iid, resolution="added kill -0 check")     # auto status=resolved, tier=archival
mk.incident_update(iid, resolved=False)                       # reopen
```

All list-adds dedupe and append. `resolution` flips status to `resolved` unless explicitly overridden by `resolved=False`.

### `incident_search(query=…, severity=…, status=…, affected=…, timeframe=…, limit=…) → list[dict]`

```python
crit_open = mk.incident_search(severity="critical", status="open")
payments  = mk.incident_search(affected="payments")
recent    = mk.incident_search(timeframe=("2026-04-01", "2026-04-20"))
```

Returns summary dicts sorted newest first. Use `mk.incident_get(id)` for the full structured view (frontmatter + sections).

### `runbook_add(pattern, steps, …) → runbook_id`

```python
rid = mk.runbook_add(
    pattern="Address already in use",
    steps=["pgrep -af gateway", "kill -0 PID", "restart guard"],
    cause="stale PID without liveness check",
    source_incident_id=iid,
    confidence=0.8,
)
```

**Upsert semantics**: adding a runbook with the same pattern merges steps, merges `source_incidents`, bumps `usage_count`, and keeps the higher of the two confidences.

### `runbook_match(symptom, …) → list[dict]`

```python
hits = mk.runbook_match("Address already in use on gateway")
# [{"id":"rb-address-already-in-use", "score":0.84, "similarity":1.0,
#   "confidence":0.82, "usage_count":3, "steps":[...], ...}]
```

Scoring: `0.6 × similarity + 0.4 × confidence`.
- Regex-like patterns are detected heuristically and evaluated with `re.search`; a regex hit yields similarity `1.0`.
- The winning runbook has its `usage_count` bumped and `confidence` nudged up (capped at `1.0`). Pass `touch=False` to observe without side effects (e.g. inside RCA).

### `incident_rca(incident_id, strategy="heuristic", include_related=True) → report`

```python
rep = mk.incident_rca(iid)
# rep["hypotheses"]           — sorted desc: confirmed > testing > rejected
# rep["suggested_runbooks"]   — runbook_match on first symptom (no touch)
# rep["related_incidents"]    — by symptom similarity + affected overlap
```

Heuristic scoring:
- `confirmed` → `0.95`
- `rejected`  → `0.05`
- `testing`   → `max(0.3, sim(symptoms, text)) + evidence_bonus`, capped `0.9`
- `evidence_bonus = min(0.25, 0.05 × len(evidence))`

`strategy="llm"` and `"both"` are reserved for a future release; v0.9 raises `ValueError`.

## Storage

```
memory/
  incidents/
    <inc-YYYY-MM-DD-slug>.md   # one file per incident
  runbooks/
    <rb-slug>.md               # one file per runbook
```

Each file is a standard MemKraft Markdown document: YAML-ish frontmatter (same parser as `tiers`/`decay`) + sectioned body. List-valued fields (`affected`, `tags`, `source_incidents`) are JSON-encoded into frontmatter strings — human-readable, `grep`-friendly, no migration needed.

Sample incident file:

```markdown
---
tier: archival
affected: ["gateway"]
detected_at: 2026-04-20T02:00:00
id: inc-2026-04-20-gateway-double-start
recorded_at: 2026-04-20T14:30:00
resolved_at: 2026-04-20T14:30:00
severity: high
source: log-analysis
status: resolved
tags: ["lifecycle-bug"]
title: Gateway double start 60K
type: incident
valid_from: 2026-04-20T02:00:00
valid_to: 2026-04-20T14:30:00
---
# Gateway double start 60K

## Symptoms
- Address already in use
- pgrep gateway returns 2 pids

## Evidence
- type: log, path: /var/log/gateway.log

## Hypotheses
- [rejected @ 2026-04-20T13:00:00] cron double run
- [confirmed @ 2026-04-20T14:30:00] stale PID file

## Resolution
added `kill -0 $PID` liveness check
```

## Migration notes

**None required.** v0.9 adds mixin methods and two directories; any v0.8.x code path works unchanged. To roll back, `pip install memkraft==0.8.4`; the `incidents/` and `runbooks/` directories are simply ignored by older versions.

## Compatibility

- Python 3.9+ (unchanged).
- Zero new runtime dependencies.
- Frontmatter round-trip uses the same parser as v0.8 tiers/decay, so existing tooling reading `.md` files is unaffected.
